# v2 Plan — Rebuild Atomic-Drill Recipe for Four Domains

Handoff document for a **fresh Claude Code session** to execute v2.0.
All facts needed to run are captured here or referenced to committed
files. Prior context: `docs/lessons.learned.md` (esp. "Training
Methodology — v1.1/v1.2 findings"), `v1/decision.md`,
`v1.1/decision.md`, `v1.2/decision.md`.

## Why this doc exists

Three training arms (v1, v1.1, v1.2) halted against v0.7-r64's gate.
The prior plan (`docs/training.process.md`) proposed warm-start as
the fix; v1.1 and v1.2 falsified that hypothesis. The actual root
cause, discovered through a synthesis-theme audit, is:

**v1's FP corpus chose compositional scenarios instead of atomic
pattern drills**, breaking the recipe that made v0.7 ship. See
`lessons.learned.md` §"v0.7's FP recipe was atomic pattern drilling."

v2 fixes the data, not the training recipe. Fresh-from-base joint
multi-task training at r=64 — exactly v0.7's approach — with a
curated dataset that keeps the proven atomic-drill records and drops
the scenario-compositional ones.

## Principles

### 1. Atomic pattern drilling for primary training signal

Each batch = ONE API surface × 30–40 minimal variations. v0.7's FP
recipe was literally this and produced FP=4.40 on 320 records
(8 batches × 40). v1's ES recipe was the same (37 different
aggregates, all implementing the Decider pattern; batch D35 was
dropped during synthesis) and produced
ES=4.50 on 1364 records. Both worked.

v1's FP regressed because it shifted to compositional scenarios
("build an HTTP handler," "hexagonal architecture") that mix 3-4
patterns per record. Advanced, but wrong for primary training.

**v2 discipline:** any new synthesis for v2 follows the pattern-spec
model from `v0.7/seeds/patterns.fp.md`. One pattern per batch. 30-40
variations via `phrasings.md`. Deep, narrow, atomic.

### 2. Reuse what worked, drop what didn't

| Source | Status | Action |
|---|---|---|
| v0.7 XState (440, atomic) | shipped, proven | **keep** |
| v0.7 FP (320, atomic 8-pattern drill) | shipped, proven | **keep** |
| v0.7 RX (240, atomic) | shipped, proven | **keep** |
| v1 ES (1364, atomic 37-aggregate Decider drill) | ES=4.50 shipped | **keep** |
| v1 FP (874, compositional scenarios) | caused FP regression | **drop** |
| v1 RX (345, mixed — needs audit) | contributed to small RX dip | **drop for v2.0** (reconsider v2.1) |
| v1 XState (702, mixed — needs audit) | marginal gain +0.10 | **drop for v2.0** (reconsider v2.1) |
| Anchors 30 × 20 reps = 600 (v1 mix, 12% ratio) | Cap dropped to 3.25-4.00 | **replace**: 30 × 8 reps = 240 (8% ratio) |

### 3. Fresh from base, joint multi-task

v1.1/v1.2 tested warm-start from v0.7 and it did not preserve prior
capability. v0.7 itself was trained fresh-from-base on joint
multi-task mix of XState + FP + RX. That is a **proven recipe on our
hardware** (RTX 5090, 32 GB VRAM, r=64 LoRA, bnb-4bit).

**v2.0 uses the same recipe**: `MODEL =
unsloth/qwen3-14b-unsloth-bnb-4bit`, fresh LoRA via
`get_peft_model(r=64, ...)`, joint shuffle across all domains, 3
epochs, lr=1e-4, seed=42. No warm-start.

### 4. One variable per arm

v1.1 and v1.2 changed base AND data simultaneously. Uninterpretable.
v2 reverts to a single-lever-per-arm discipline:

- **v2.0** (this plan): test whether v0.7's recipe works on 4 domains
  with v0.7's data + v1's atomic-drill ES + reduced anchors. Single
  variable: adding ES to the proven 3-domain recipe.
- **v2.1** (only if v2.0 ships): test if v1's XState-extension
  batches (702 records) add value without regressing. Single
  variable: XState depth on top of v2.0.
- **v2.2** (only if v2.1 ships): same for RX extension (345 records).
- **v2.3+** (future): rebuild missing FP atomic patterns (v0.7
  covered 8 of 15 spec'd patterns; 7 remain untrained) — adds FP
  breadth without introducing compositional contamination.

## v2.0 dataset recipe

### Composition

| Source | Domain | Records | Notes |
|---|---|---|---|
| `v0.6/data/synth.verified.jsonl` | xstate | 440 | v0.6 XState, atomic |
| `v0.7/data/synth.fp.batch-{A..H}.jsonl` | fp | 320 | v0.7 FP, 8 atomic-drill batches |

| New: 30 anchors × 8 reps | capability | 240 | v0.7's 12 + v1's 18 new, 8% ratio |
| **Total** | | **~2604** | |

Dedup by hash (prompt+response) — should be near-zero collisions
because each source is already unique.

Anchor reps dropped from v1's 20 → 8 per v1/decision.md's
recommendation ("8–10% anchor ratio, 20+ unique anchors, 8–12 reps
each"). 240 anchors / 2604 = 9.2% ratio.

- Remove v1 FP batch loading (drop `V1_FP`)
- Remove v1 RX batch loading (drop `V1_RX`)
- Remove v1 XState batch loading (drop `V1_XSTATE`)
- Add v0.6 XState load (`v0.6/data/synth.verified.jsonl`)
- Add v0.7 FP + RX batch loads (`v0.7/data/synth.{fp,rx}.batch-*.jsonl`)
- Keep v1 ES batch load (already atomic drill)
- Change anchor reps: `ANCHOR_REPS = 8` (from 20)
- Keep all 30 anchors (don't filter)

## Predicted outcomes

Based on the hypothesis "atomic-drill data + fresh-base + joint
multi-task reproduces v0.7's success at 4-domain scale":

| Domain | v0.7 | v2.0 predicted | Rationale |
|---|---|---|---|
| XState | 4.10 | 4.10 ± 0.2 | Same 440 records as v0.7; maybe tiny lift from multi-task cross-training |
| FP | 4.40 | **4.30 – 4.50** | Same 320 atomic records; noise-limited around v0.7 |
| Reactive | 4.80 | 4.60 – 4.80 | Same 240 records; possible small dip from 4-domain capacity sharing at r=64 |
| Event Sourcing | n/a | **4.30 – 4.60** | 1364 atomic records proved v1 value, but r=64 serves 4 domains now |
| Capability | 4.25 | 4.10 – 4.30 | 240 records at 8% ratio, same 12 anchors + 18 new |
| **Overall avg** | **4.10** | **4.15 – 4.45** | |

**Ship target**: overall ≥ 4.20 AND no trained-domain regression
vs v0.7 beyond the gate tolerances. If hit, v2.0 becomes the new
daily driver and v0.7-r64 rolls off rotation.

## Failure modes

| Observation | Interpretation | Next move |
|---|---|---|
| FP ≥ 4.30 | Recipe scales from 3 to 4 domains. | Run v2.1 (add XState extension). |
| FP 4.00–4.29 | Marginal regression within noise. Try to interpret. | Grader recalibrate against v0.7; if real regression, r=64 may be capacity-limited across 4 domains → test r=128 before shipping. |
| FP < 4.00 | Unexpected — something in the simpler recipe broke. | Diff v2.0 data against v0.7+v1.ES. Likely a dedup mishap or a stale batch. |
| RX < 4.50 | Capacity starvation suspected (r=64 for 4 domains). | Test r=128 as v2.0b variant. |
| ES < 4.00 | Joint multi-task diluted the ES learning. | Try r=128 OR accept and ship as a 3-domain model. |
| Cap < 4.00 | Anchors at 8% too low for the new 18 unseen ones. | Adjust to 10% (30 × 10 = 300) in v2.1. |

discipline. Cost: ~6-8h of subagent synthesis orchestration.

## What v2 is NOT

- **Not a warm-start experiment.** We tested warm-start in v1.1/v1.2
  and it did not preserve capability. Going back to fresh-from-base
  with proven recipe.
- **Not a LoRA stacking experiment.** Stacking is the architectural
  alternative for capability preservation but requires ~1 day of
  engineering (adapter recovery, PEFT multi-adapter plumbing,
  GGUF-merge rework). Defer until v2.0 and its fallbacks run.
- **Not new data synthesis.** v2.0 uses only committed records.
  Synthesis work (v2.3+) is deferred until the reuse-only arm ships.
- **Not a regex tightening.** The verifier stays as v0.7/verify.py.
  Tightening it (from Phase 3 of training.process.md) is now
  understood to be a band-aid on synthesis-discipline issues — the
  real fix is upstream theme selection, which v2.0 controls by
  data sourcing.
