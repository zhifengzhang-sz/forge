# v0.6 — Real Fine-Tuning Test, with Claude as Teacher

This document is the handoff for a **fresh Claude Code session**. Do not assume
the prior session's conversation is available. All facts needed to execute
v0.6 are captured here or referenced via files on disk.

## Why v0.6 exists

v0 ran fine-tuning at 50-example scale. That scale is below the published
floor (~1k–10k) for using LoRA to overcome a model's syntactic priors. v0
therefore did not test whether fine-tuning *can* close the gap — it tested
whether it can at toy scale (no, with mild signal that extracted-usage data
is directionally better than hand-curation).

v0 also surfaced a teacher: **Claude Opus 4.7 scored 4.95 / 5.00 across the
24-prompt eval, including a perfect 5.00 on XState** (see `v0/results/claude_opus.json`
and `v0/grading/grade_claude_opus.py`). With a frontier-quality teacher available,
the data-collection bottleneck dissolves: synthesise pairs from Claude, verify
each with `tsc --strict --noEmit`, train at proper scale.

The honest framing of the project's premise:

> Can a 14B local model, properly fine-tuned, close most of the gap to a frontier
> model on a narrow domain?

If **yes** (qwen3:14b reaches 3.5–4.5 on XState), the project's premise is
validated and v1 is "scale across all 4 domains". If **no** (qwen3 stays at
2.0–2.5 even with proper data), the conclusion is much stronger than v0's:
LoRA fine-tuning is essentially a useless tool for narrow-domain syntactic
adaptation, and the field's reputation depends on the result.

v0.6 is the experiment that earns that conclusion either way.

## v0.6 plan

### Phase A — Synthesise

Generate 5000 candidate (instruction, completion) pairs for XState v5 by calling
Claude Opus 4.7 via the Anthropic API.

**Inputs to the synthesis prompt (all live on disk):**

- The 46 extracted examples (real machines from `repos/xstate/examples/`) as
  pattern seeds.
- The official XState v5 docs (clone `https://github.com/statelyai/xstate` and
  reference the `docs/` subtree, or scrape the website at synthesis time).
- A diverse list of *instruction phrasings* derived from real prompts (see
  `v0/eval/v0.json` for shape, but generate variations — do NOT use the eval
  prompts as training data).

**Method:**

1. For each of ~30 XState patterns (counter, toggle, fetch, multi-step form,
   parallel states, hierarchical states, fromCallback, fromObservable, spawn
   child, sendTo, emit, enqueueActions, after delays, always transitions,
   guards in setup, output from final, custom actor logic, parent-child
   communication, reentry/internal transitions, snapshot persistence, …):
   - Generate ~150 instruction variations (different phrasings, scopes,
     contexts: "Build a …", "Fix this v4 …", "Add … to this machine", "What's
     the v5 way to …").
   - For each instruction, ask Claude Opus 4.7 to produce a complete v5
     answer in the same style as the v0 baseline (see `v0/results/claude_opus.json`).
2. Save raw output as `v0.6/data/synth.raw.jsonl`.

**Cost estimate:** 5000 calls × ~700 input tokens × ~600 output tokens at
Opus 4.7 pricing → roughly $30–80. Confirm budget with the user before
running.

### Phase B — Verify

Reject pairs whose completion fails any of these gates:

1. **Compiles**: write the completion to `/tmp/probe-NN.ts` in a temp
   directory with `tsconfig.json` set to `strict: true`, `target: es2022`,
   `moduleResolution: bundler`, `types: ["xstate"]`. Install `xstate@5` and
   `@types/node` in that temp dir. Run `tsc --noEmit /tmp/probe-NN.ts`.
   Drop on non-zero exit.
2. **Uses v5 idioms**: regex must match `setup\(`. Regex must NOT match
   `cond:`, `\binterpret\(`, `services:\s*\{`, `\(ctx, event\)\s*=>`,
   `Machine\(`.
3. **Length sanity**: completion is between 150 and 6000 chars. Drop outliers.

Realistic survival rate: **40–70%** based on Claude's v0 baseline quality
plus the v5 strictness. Surviving count target: **2.5k–3.5k pairs** going
into Phase C.

Save survivors to `v0.6/data/synth.verified.jsonl`.

### Phase C — Train

Use `v0/train_v0.py` with three changes from v0:

| Hyper-param | v0 | v0.6 | Why |
| --- | --- | --- | --- |
| LoRA rank `r` | 16 | **32 or 64** | Capacity for syntactic shift; r=16 was forgetting-protective for 50 examples — at 3k examples, more capacity is appropriate. |
| Epochs | 2 | **3** | More steps to actually internalise the new syntax. |
| Mix-in | none | **+10% capability anchors** | Re-use the 8 capability anchors from `v0/data/xstate_curated.jsonl` to preserve tool calling / general reasoning. |

Output: `v0.6/gguf/qwen3-14b.Q4_K_M.gguf` plus auto-generated Modelfile.

### Phase D — Evaluate

1. `ollama create ts-forge-v0.6 -f v0.6/gguf/Modelfile` (path from train output).
2. `python v0/run_eval.py --model ts-forge-v0.6 --arm v0.6` — produces
   `v0/results/v0.6.raw.json`.
3. `python v0/tool_call_smoke.py --model ts-forge-v0.6 --arm v0.6` — produces
   `v0/results/v0.6.toolcall.json`. Must hit ≥ 2/3 to claim outcome (B) is still
   ruled out.
4. Grade via a fresh-subagent pattern (see `v0/grading/grade_claude_opus.py`
   for how to dispatch a subagent for blinded grading). Save the grading
   script to `v0/grading/grade_v0.6.py`.

### Phase E — Decide

Compare against the v0 baseline + Claude:

| If v0.6 xstate avg lands at … | Conclusion |
| --- | --- |
| ≥ 4.0 | Premise validated. Move to v1 (scale to all 4 domains, same recipe). |
| 3.0 – 4.0 | Partial win. Worth a v0.7 with more data / different rank. |
| < 3.0 | LoRA fine-tuning is not closing the gap on syntactic correction at this base model + domain combination. Strong conclusion: pivot to RAG over v5 docs as the production approach for this project's stated goal, and treat fine-tuning as research material rather than product strategy. |

Whatever the outcome, write `v0.6/decision.md` documenting the result alongside
the same numbers table format as `v0/decision.md`.

## What the fresh session needs from the user

1. **`ANTHROPIC_API_KEY`** in the environment (the user said in the v0 session
   they'd rather not pay per-call API; if they hold to that, v0.6 cannot
   proceed via API and the alternative is to drive synthesis through Claude
   Code subagents — slower and bandwidth-bounded by the user's plan).
2. **Budget approval** for ~$30–80 in Anthropic API spend.
3. **Decision on r=32 vs r=64**. Recommend starting with r=32 as a safer
   default; jump to r=64 only if v0.6 lands in the 3.0–4.0 range and we
   want to push further.

## What v0.6 must NOT do

- **Do not modify `v0/eval/v0.json`.** It is the apples-to-apples comparator.
- **Do not put eval prompts into training data.** Even paraphrases — the eval
  set is the contamination boundary.
- **Do not skip the tool-call canary.** It is the only check on the (B)
  outcome.
- **Do not let Claude grade v0.6 inside the same session that ran the
  synthesis.** Use a fresh subagent for grading, same pattern as
  `v0/results/claude_opus.json` was produced.

## Methodology debt v0.6 should also address

The v0 grading was non-blinded — the same Claude that ran the experiments
also assigned scores. v0.6 should at least dispatch a fresh subagent per
arm to re-grade, and record both the original grades and the independent
grades so disagreements are visible.

If both arms rate the v0.6 fine-tune at ≥ 4.0 on XState, that's a stronger
signal than one grader alone.
