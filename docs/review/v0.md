# v0: The "Do We Even Need This?" Spike

One-day exercise. One question it answers: on *your actual TypeScript tasks*, does fine-tuning meaningfully beat the base Qwen3-Coder-30B? Everything else follows from the answer.

## Phase 0 — Write the eval set first (30-45 min)

Before you touch a model, write the test. This is the only artifact from v0 that has lasting value regardless of outcome, so spend real time on it.

15-20 prompts that represent actual TypeScript work you'd do. Hand-written, not extracted from any repo. Roughly 4-5 per domain. Each one with concrete success criteria:

```json
{
  "id": "xstate-01",
  "domain": "xstate",
  "prompt": "Create an XState v5 machine using setup() for a multi-step form with steps 'email', 'password', 'confirm'. Include fromPromise for email validation and typed events.",
  "must_have": ["setup(", "fromPromise", "assign(", "types:"],
  "must_not_have": [": any", ".send("]
}
```

Then add **3-5 "capability regression" prompts** that have nothing to do with your domains:

- "Write a Python function to compute Fibonacci numbers"  (still multilingual?)
- "Use the Read tool to look at package.json, then summarize"  (still does tool calls?)
- "What are the tradeoffs between REST and GraphQL?"  (still reasons/explains?)

These are your canary for catastrophic forgetting. They're the reason you'd know whether a fine-tuned model breaks Claude Code integration before you invest in building the full pipeline.

## Phase 1 — Baseline the base model (30-60 min)

```bash
curl -fsSL https://ollama.com/install.sh | sh
ollama pull qwen3-coder:30b
# Run every prompt from eval/v0.json; grade 1-5 manually
```

**Decision gate 1 — this is the most valuable step in the whole plan:**

| Average score on domain prompts | What to do |
|---|---|
| ≥ 4.0 | Stop. Base model is good enough. Use `ollama launch claude --model qwen3-coder:30b`. Project complete. |
| 2.5 - 4.0 | Fine-tuning is plausibly worth it. Proceed. |
| ≤ 2.5 | Base model is too weak; 1K examples of LoRA won't rescue it. You need a different strategy (continued pretraining, more data, bigger model), not your current plan. |

Most people skip this gate and train anyway. Don't.

## Phase 2 — Hand-curate minimal training data (2-3 hours)

Pick **one domain first.** I'd pick XState because it's the least well-represented in most pretraining data, so the signal-to-noise ratio of fine-tuning is highest.

Write 50-75 `(instruction, completion)` pairs by hand. Not extracted. Not templated. Format as messages JSONL. Approximate mix:

- ~40 "write this pattern" examples: setup(), fromPromise, assign with typed context, guarded transitions, parent-child actors, invoke with input
- ~10 "fix this code" examples: input uses v4 API, output uses v5 
- ~10 **capability-anchor examples**: tool-call transcripts, Python snippets, general explanations, with good completions. This is your forgetting mitigation — without it, you will lose Claude Code tool use.

Yes this is tedious. But 50 hand-curated examples will outperform 1,100 template-generated ones, and you find out in a day whether the approach itself works. If it works with 50 hand-curated, it'll work better with 500 hand-curated. If it doesn't work with 50 hand-curated, 500 template-generated won't save you.

## Phase 3 — Minimum-viable train (45 min, mostly waiting)

```bash
docker run --gpus all -it -v $(pwd):/workspace unsloth/unsloth bash
```

Then inside, `train_v0.py` is ~50 lines — no pipeline, no orchestrator:

```python
from unsloth import FastLanguageModel
from trl import SFTTrainer, SFTConfig
from datasets import load_dataset

# verify exact slug on unsloth's HF page; MoE 4bit path has known quirks
MODEL = "unsloth/Qwen3-Coder-30B-A3B-Instruct-unsloth-bnb-4bit"

model, tok = FastLanguageModel.from_pretrained(
    MODEL, max_seq_length=4096, load_in_4bit=True,
)
model = FastLanguageModel.get_peft_model(
    model,
    r=16, lora_alpha=16, lora_dropout=0,   # r=16 not 32: forgetting protection
    target_modules=["q_proj","k_proj","v_proj","o_proj",
                    "gate_proj","up_proj","down_proj"],
    use_gradient_checkpointing="unsloth",
)

ds = load_dataset("json", data_files="xstate_v0.jsonl", split="train")
ds = ds.map(lambda x: {"text": tok.apply_chat_template(x["messages"], tokenize=False)})

SFTTrainer(
    model=model, tokenizer=tok, train_dataset=ds,
    args=SFTConfig(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=4,
        num_train_epochs=2,          # not 3 — forgetting risk on small data
        learning_rate=1e-4,          # not 2e-4 — forgetting risk
        bf16=True, logging_steps=5,
        output_dir="out", save_strategy="no",
    ),
).train()

model.save_pretrained_gguf("gguf_v0", tok, quantization_method="q4_k_m")
```

Three protective defaults vs your current `train.py`: r=16 not 32, lr=1e-4 not 2e-4, 2 epochs not 3. Training ~75 examples takes 5-10 minutes.

## Phase 5 — The decision

Three outcomes, each with a clear next step:

**(A) Fine-tuned beats base on domain AND holds capability prompts.** The approach works. Scale up the existing pipeline with v0's lessons baked in (lower rank, capability anchors, usage-not-internals data, LLM-generated instructions instead of templates). Nothing you built before is wasted.

**(B) Fine-tuned beats base on domain, but capability prompts regress.** Catastrophic forgetting confirmed. Tool calling is likely broken; Claude Code integration won't work as designed. You now know this on day 1 instead of week 4. Options: increase capability-anchor ratio (50% anchor data, not 15%), drop rank to 8, drop to 1 epoch, or accept the model is domain-only and won't work in Claude Code (revise the goal).

**(C) Fine-tuned doesn't beat base, or is worse.** Either 50 examples wasn't enough (try 150 hand-curated before concluding), or the extraction-based approach was always going to be wrong for your actual use case. Reconsider the premise.

---

## Why this beats what you have today

- **Total new code:** ~100 lines. The existing `extract_pipeline.py`, `lib/`, `app/`, and agents stack stays untouched.
- **Total API cost:** $0. Manual grading on 20 prompts takes 30 minutes.
- **Total training time:** 5-10 min.
- **End-of-day knowledge:** which of (A), (B), (C) you're in — which determines whether the existing pipeline should be scaled up, retargeted, or retired.

The specific thing v0 protects you from is spending another week on the pipeline only to discover outcome (B) — that fine-tuning breaks the Claude Code integration goal you built everything around. That's the failure mode that's cheap to detect now and expensive to detect later.

One thing to be honest about: if the result is (A), you'll have spent a day that your existing pipeline could arguably have replaced. If the result is (B) or (C), you've saved weeks. The expected-value math works out because (B) and (C) are more likely than people expect given how narrow your training data is and how much you're asking of a 1K-example LoRA.
