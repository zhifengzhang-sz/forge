# v1 — Scale to All Four Domains, 5000 Pairs

This document is the handoff for a **fresh Claude Code session** to execute v1.
All facts needed are captured here or referenced via committed files. For
prior context see `v0/decision.md`, `v0.6/decision.md`, and `v0.7/decision.md`.

## Why v1 exists

v0.7 validated the recipe on three domains (XState, FP, RX). Every
predeclared threshold cleared: XState 4.10, FP 4.40, Reactive 4.80, tool
calls 3/3. Winner was r=64 with 4.10 domain average.

Unaddressed at v0.7:
- **Event Sourcing stayed at 3.00–3.10** because we never trained on it.
  Same story across v0, v0.6, and v0.7 — ES has been the untouched domain
  throughout.
- **Scale plateaued at ~1000 pairs.** The ~1k LoRA floor cited in v0.6's
  plan was treated as a floor, not as a target. v1 tests whether 5000
  pairs pushes the trained domains past 4.5 toward the Opus ceiling at 4.95.

v1 is the scale-up run that adds ES to the mix and tests whether the
recipe continues working at 5× volume.

## Predeclared outcomes (same structure as v0.7)

| Condition | Conclusion |
|---|---|
| XState ≥ 4.5 AND FP ≥ 4.0 AND Reactive ≥ 4.5 AND ES ≥ 3.8 | **Premise exceeded. Ship as daily driver.** |
| All domains trained ≥ 3.5 but below the above thresholds | Solid v1. Consider quantization / prompt engineering rather than more data. |
| XState or any trained domain regresses below v0.7 levels | Investigate — scaling may have destabilized training. Possible over-fitting. |
| ES stays below 3.5 despite training | ES-specific patterns may need different approach (different seed style, more complex examples). |

Tool calls ≥ 2/3 required in every branch. Uncertainty bands (two-grader)
required as in v0.7.

## Scope (subject to plan-eng-review)

~5000 verified training pairs, split:

| Domain | Count | Share | Notes |
|---|---|---|---|
| XState | 1000 | 20% | Reuse v0.6's 440 + v0.7 coverage doesn't exist as separate pairs (v0.7 only trained on FP/RX + reused XState). Need ~560 more new XState depth pairs. |
| fp-ts / Effect-TS | 1200 | 24% | v0.7 produced 320; need ~880 more. Expand to v3 Stream patterns, Exit types, Brand-refined. |
| RxJS | 1000 | 20% | v0.7 produced 240; need ~760 more. Expand to higher-order observables, multicasting variants, error isolation. |
| Event Sourcing | 1500 | 30% | **All new.** Target ~15 patterns: Decider interface, pure evolve/decide, projections, snapshotting with version tags, optimistic concurrency with expectedRevision, subscription from checkpoint, event store abstraction, typed command/event/state discriminated unions, saga patterns, choreography vs orchestration, upcasting, read model rebuild, CQRS boundary, integration events, outbox pattern. |
| Capability anchors (mixed in at 5%) | ~260 | 5% | Same 12 unique anchors from v0.7 × ~22 reps each. Repetition count is high — consider expanding to 20 unique anchors to keep per-anchor reps at ~13. |

**Total:** ~4960 training records.

## Plan

### Phase A — Seeds

Extend `v1/seeds/` from v0.7 seeds:
1. Copy `v0.7/seeds/patterns.xstate.md`, `patterns.fp.md`, `patterns.rx.md`,
   `phrasings.md`, `reference_examples.xstate.jsonl`, `anchors.jsonl`.
2. Write `v1/seeds/patterns.es.md` — **new**. ~15 ES patterns covering
   the bullets in the scope table above. Each entry matches the style
   of v0.7/seeds/patterns.fp.md (one-line summary + key APIs + must-haves).
3. Write `v1/seeds/reference_examples.es.jsonl` — 20-30 real ES examples
   from `repos/EventSourcing.NodeJS/` (mechanically extracted like v0.6 did
   for XState). Keep them as reference for synthesis subagents.
4. Consider expanding `anchors.jsonl` from 12 to 20 anchors to reduce
   per-anchor repetition at 5% × 5000 records.

### Phase B — Verifier (per-domain, extended to ES)

Fork `v0.7/verify.py` to `v1/verify.py`. Add ES domain:

| Gate | ES |
|---|---|
| Compile | `tsc --strict --noEmit` with deps: `@types/node` (pure TypeScript). Optional: an `@eventstore/db-client` or similar if patterns reference it. Keep the dependency set minimal. |
| Idiom MUST-MATCH (any of) | `evolve\(`, `decide\(`, `Decider`, `CommandHandler`, `EventStore`, `Snapshot`, `expectedRevision`, `appendToStream`, `readStream` |
| Idiom MUST-NOT-MATCH | `from ['"]xstate['"]`, `setup\(`, `Effect\.gen` (shape guards to prevent other-domain leakage back into ES) |
| Length | 150–6000 chars |

XState / FP / RX gates: unchanged from v0.7.

### Phase C — Synthesis

Same subagent-driven pattern. Wave-budget proposal (subject to plan-eng-review
adjustment):

| Wave | Domain | Agents | Pairs | Cumulative |
|---|---|---|---|---|
| A | XState depth | 14 | 560 | 1000 (w/ v0.6 reuse) |
| B | FP expand | 22 | 880 | 1200 |
| C | RX expand | 19 | 760 | 1000 |
| D | ES fresh | 38 | 1500 | 1500 |

Total subagent dispatches: ~93. At ~5–15 min per wave of 7 parallel, wall
time estimate: **4–6 h**. The plan-eng-review for v1 should revise this

Alternative: drop scope to **3000 pairs** to halve synthesis time while still
≥3× v0.7 volume. Plan-eng-review is the place to decide this tradeoff.

### Phase D — Verify + merge

### Phase E — Train

v1/train.py = fork of v0.7/train.py. Defaults:
- Rank **r=64** (locked from v0.7 winner).
- Epochs **3** (locked).
- LR **1e-4** (locked).
- Anchor ratio **5%** (locked).
- Seed 42.

Consider running r=64 AND r=128 sequentially if GPU time permits and v0.8
is interested in whether higher rank helps at 5× data. Deferred to
plan-eng-review.

Same pre-flight checks:
- `systemctl is-active nvidia-pl.service` → active
- `nvidia-smi --query-gpu=power.limit --format=csv,noheader` → 500 W
- 30 s sleep between `save_pretrained_gguf` and `ollama create`.

Training wall time estimate (based on v0.7: 15 min per arm at 1053 records):
5000 records × 3 epochs / batch size 8 ≈ 1875 steps. Assuming 0.5 s/step
on the 5090 → ~15–20 min training + 7 min GGUF conversion = ~25 min per arm.

### Phase F — Evaluate

## Acceptance criteria for the v1 plan-eng-review

This file is a first draft. Before executing v1, run `/plan-eng-review` on
this doc and resolve:

1. **Is ~5000 the right scale?** Or should v1 be conservative at 3000 first?
2. **Is the 20/24/20/30/5 proportion defensible?** Especially: ES at 30% —
   is that too much for a domain that's getting first-pass treatment?
3. **Should r=128 get an A/B slot?** Or is r=64 locked?
4. **Capability anchors at 5% × 5000 = 250 records from 12 unique.** 20-ish
   reps per anchor. Memorization risk assessment.
5. **Should eval expand?** At some point n=5 per domain becomes the
   bottleneck. Trade-off: apples-to-apples with v0-v0.7 vs statistical
   precision for claims like "XState 4.7".
6. **Grader pool** — same two graders as v0.7 or does v1 benefit from
   a third? Grader disagreement on v0.7 was 0.25 mean (tight) — might not
   need more.
7. **Process for ES pattern seeds** — hand-write vs extract from
   `repos/EventSourcing.NodeJS`. v0.6 tried both for XState; the extracted
   approach worked better directionally.

## What v1 must NOT do

- Modify `v0/eval/v0.json` (comparator).
- Reuse any training pair in the eval set (contamination).
- Skip the tool-call canary or power-cap pre-flight.
- Dispatch grading subagents that can read each other's output files.
- Let training and ollama-create overlap (no 30s sleep = risk of PSU trip,
  per v0.6 incident).
