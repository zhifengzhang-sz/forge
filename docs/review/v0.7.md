# v0.7 — Multi-Domain Fine-Tuning, Fixing the XState-Leakage Regression

This document is the handoff for a **fresh Claude Code session** to execute.
All facts needed to execute v0.7 are captured here or referenced via files
on disk. For v0's history see `v0/decision.md`; for v0.6's see
`v0.6/decision.md`.

## Why v0.7 exists

v0.6 produced a partial win on XState (1.60 → 3.60, plan's 3.0–4.0 band)
with a surprising side effect: **fp-ts / RxJS regressed** (fp 2.60 → 2.60
neutral; reactive 4.00 → 2.80 regression), because the training set was
100% XState and the model learned to reach for XState shapes in adjacent
TypeScript-library prompts. Specific failures on the v0 eval:

- `fp-02` (Effect-TS Layer composition): produced an XState machine.
- `fp-05` (Effect-TS Context.Tag / Layer.merge): produced `setup({...})`.
- `rx-02` (RxJS combineLatest): imported from `xstate`.

The v0.6 decision named this the **XState-leakage** failure mode. v0.7 is
the experiment that tests whether broadening the training set fixes it
while continuing to push XState upward toward the ≥ 4.0 premise-validated
threshold.

## Predeclared outcomes

| Condition | Conclusion |
|---|---|
| XState ≥ 4.0 **and** FP ≥ 3.5 **and** Reactive ≥ 3.5 **and** tool calls ≥ 2/3 | Premise validated. Move to v1 (scale across all four domains). |
| XState ≥ 4.0 **but** FP or Reactive < 3.5 | XState premise works; domain-balance in training needs more work. v0.8 with different mix. |
| XState < 4.0 **and** FP ≥ 3.5 **and** Reactive ≥ 3.5 | Multi-domain training fixes leakage but doesn't accelerate XState. Investigate rank or scale. |
| XState < 4.0 **and** (FP or Reactive) < 3.5 | Multi-domain training is not sufficient at this scale. Reconsider recipe. |

Tool calls must stay ≥ 2/3 in every branch (otherwise outcome (B) re-opens).

## Scope

~1400 verified training pairs, split:

| Domain | Count | Share | Rationale |
|---|---|---|---|
| XState (v0.6 reused + new depth) | 650 | 47% | v0.6 pairs at 440 + ~210 new depth-variety. Avoids re-synthesising what already worked. |
| fp-ts / Effect-TS | 400 | 29% | Enough to establish library presence; zero base rate in v0.6. |
| RxJS | 300 | 21% | Enough to restore reactive signal above base's 4.00. |
| Capability anchors (mixed in) | 50 | 4% | Unchanged from v0.6; preserves tool calling + general reasoning. |

Total: ~1400. Slightly above the published ~1k LoRA floor the original v0.6
plan cited. Deliberately not aiming for 5000 — v0.6 demonstrated meaningful
signal at 440, so "modest sufficient count + multi-domain" is a better
experiment than "massive single-domain."

## Plan

### Phase A — Seeds

Extend the seed artifacts under `v0.7/seeds/`:

1. `v0.7/seeds/patterns.xstate.md` — copy of `v0.6/seeds/patterns.md` (30
   patterns, unchanged) so v0.7 is self-contained.
2. `v0.7/seeds/patterns.fp.md` — **new**, ~15 fp-ts / Effect-TS patterns.
   Must include: Effect.gen with yield*, Layer.succeed / Layer.effect /
   Layer.merge / Layer.provide, Context.GenericTag, Effect.catchTag, tagged
   errors (class with `_tag`), Schedule retry, Effect.all, Option chain
   (O.fromNullable, O.chain), Either chain (E.map, E.chainW, E.mapLeft),
   TaskEither.tryCatch with error narrowing, pipe composition, branded
   types, Validation accumulation, Effect.scoped for resource acquire/release.
3. `v0.7/seeds/patterns.rx.md` — **new**, ~10 RxJS patterns. Must include:
   debounceTime + distinctUntilChanged + switchMap (typeahead), combineLatest
   with typed ViewModel, forkJoin, mergeMap with concurrency limit,
   BehaviorSubject (with getValue() and late-subscriber semantics),
   ReplaySubject, tap with next/error/complete object form, retry with
   backoff, custom operator via pipe(), shareReplay for caching.
4. `v0.7/seeds/phrasings.md` — copy of `v0.6/seeds/phrasings.md` (10
   templates, unchanged — they are library-agnostic).

No eval-prompt text from `v0/eval/v0.json` may appear in seeds, same
contamination rule as v0.6.

### Phase B — Verifier (per-domain)

Extend `v0.6/verify.py` into `v0.7/verify.py` with a per-domain gate
architecture. The verifier receives a `--domain {xstate|fp|rx}` flag (or
reads `domain` from each JSONL record). Per-domain behaviour:

| Gate | XState | fp | rx |
|---|---|---|---|
| Compile | `tsc --strict --noEmit`, deps: `xstate@^5`, `@types/node` | `tsc --strict --noEmit`, deps: `fp-ts@^2`, `effect@^3`, `@types/node` | `tsc --strict --noEmit`, deps: `rxjs@^7`, `@types/node` |
| Idiom MUST-MATCH | `setup\(` | one of: `Effect\.gen`, `pipe\(`, `Layer\.`, `E\.chain`, `O\.chain`, `TE\.chain` | one of: `pipe\(`, `switchMap\(`, `combineLatest\(`, `mergeMap\(`, `forkJoin\(`, `BehaviorSubject` |
| Idiom MUST-NOT-MATCH | `\bcond:`, `\binterpret\(`, `\bservices:\s*\{`, `\(ctx,\s*event\)\s*=>`, `(?<![A-Za-z])Machine\(` | `from ['"]xstate['"]` (XState-leakage guard — rejects training pairs that reach for XState when the prompt asks for fp); `Effect\.service\(` (deprecated) | `from ['"]xstate['"]`; `\.do\(` (pre-v7 operator style) |
| Length | 150–6000 chars | 150–6000 chars | 150–6000 chars |

Probe dir reuse: one per domain (`v0.7/.verify-tmp/xstate`, `.../fp`,
`.../rx`), each with its own `package.json` so the npm install cost is paid
once per domain.

### Phase C — Synthesis

Same subagent-driven pattern as v0.6 — no Anthropic API calls.

**Wave budget (7 parallel max per wave, 40 pairs per agent):**

| Wave | Domain | Agents | New pairs | Rationale |
|---|---|---|---|---|
| A — Reuse | XState | 0 | 0 (reuse 440 from v0.6) | Save cost. |
| B — XState depth | XState | 5 | 200 | +3 patterns each, focused on areas v0.6 undersampled per the synthesis-wave distribution. |
| C — FP1 | fp-ts / Effect | 5 | 200 | Covers ~10 of the 15 patterns. |
| D — FP2 | fp-ts / Effect | 5 | 200 | Remaining 5 patterns + depth. |
| E — RX | RxJS | 4 | 160 | Covers all 10 patterns. |
| F — fill | mixed | 3–5 | 120–200 | Top up any domain that missed 100% survival. |

Total synthesis dispatches: ~27 agents. At 15 min average per wave of 7
parallel (v0.6 wave-2 precedent), walltime ~75 min. Plus iteration overhead
(some fp/rx pairs will iterate on tsc errors), realistic estimate 90–120 min.

Each synthesis subagent prompt is parameterised by (domain, pattern list,
target count) — a single template adapts for all three domains. Subagents
self-verify against `v0.7/verify.py --domain {domain}` before writing their
final batch file, same as v0.6 wave-2.

### Phase D — Verify + merge

Run `v0.7/verify.py` per domain on each batch. Merge survivors into
`v0.7/data/synth.verified.jsonl` (with a `domain` field per record for

Expected survival rate based on v0.6 experience: XState ~100%, FP ~85%
(some Effect API surface is narrow and tsc will catch shape errors), RX

## Risks / open questions for reviewer

These are the places I think I could be wrong:

1. **Domain proportion.** 47% XState / 29% FP / 21% RX might be wrong. An
   alternative view: since XState already works at v0.6 levels, reducing it
   to 30% and pushing FP/RX higher (~35% each) might fix the leakage faster
   without losing XState. I don't have strong evidence either way.

2. **Reusing v0.6 pairs.** If the fp/rx failure mode is "model has a strong
   XState prior," reusing v0.6's 440 XState-only pairs reinforces that prior.
   An alternative is to **re-synthesise XState pairs under the multi-domain
   prompt** so the synthesis signal is inherently domain-diverse from the
   start. Costs more subagent time.

3. **Verifier XState-leakage guard.** Rejecting fp/rx pairs that import from
   `xstate` is the right rejection at training time, but can we trust the
   verifier's gate set to catch all forms of XState-shaped code in an
   Effect-TS answer? A synthesis subagent could produce `setup`-shaped code
   in a file that happens to import from `effect` — the regex would miss it.
   Mitigation: the MUST-MATCH for fp includes Effect-specific tokens, so the
   synth would fail that gate instead. But it's not airtight.

4. **Single-arm grading bias.** The v0.6 grader was asked to "grade on
   quality alone, blinded to arm" but knew it was grading one arm. For v0.7,
   mixing v0.6 and v0.7 responses in a single blinded grading pass would
   give cleaner comparisons. Not implemented in the plan above — worth
   discussing.

5. **r=32 vs r=64.** Locked r=32 from v0.6 validation, but v0.7 pushes more
   aggressive syntactic shifts across three domains. r=64 may be worth a
   parallel arm if GPU time permits.

6. **Capability anchors count may need to scale.** 10% of 1400 = 140 anchor
   records, from 8 unique sources → 17 repetitions each. v0.6 had 6
   repetitions each. Unclear whether more repetition helps or overfits.
