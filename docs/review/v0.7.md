# v0.7 — Multi-Domain Fine-Tuning, Fixing the XState-Leakage Regression

This document is the handoff for a **fresh Claude Code session** to execute.
All facts needed to execute v0.7 are captured here or referenced via files
on disk. For v0's history see `v0/decision.md`; for v0.6's see
`v0.6/decision.md`.

## Why v0.7 exists

v0.6 produced a partial win on XState (1.60 → 3.60, plan's 3.0–4.0 band)
with a surprising side effect: **fp-ts / RxJS regressed** (fp 2.60 → 2.60
neutral; reactive 4.00 → 2.80 regression), because the training set was
100% XState and the model learned to reach for XState shapes in adjacent
TypeScript-library prompts. Specific failures on the v0 eval:

- `fp-02` (Effect-TS Layer composition): produced an XState machine.
- `fp-05` (Effect-TS Context.Tag / Layer.merge): produced `setup({...})`.
- `rx-02` (RxJS combineLatest): imported from `xstate`.

The v0.6 decision named this the **XState-leakage** failure mode. v0.7 is
the experiment that tests whether broadening the training set fixes it
while continuing to push XState upward toward the ≥ 4.0 premise-validated
threshold.

## Predeclared outcomes (post-review)

Per plan-eng-review issue #3, added explicit regression branches (XState <
3.60 baseline) because rebalancing to 32% XState makes regression a plausible
outcome. Also report scores with uncertainty bands from two-grader
disagreement (issue #8); "borderline" is ±0.3 around any threshold.

| Condition | Conclusion |
|---|---|
| XState ≥ 4.0 **and** FP ≥ 3.5 **and** Reactive ≥ 3.5 | Premise validated. Move to v1. |
| XState ≥ 4.0 **but** FP or Reactive < 3.5 | XState premise works; domain-balance needs more work. v0.8 with different mix. |
| 3.60 ≤ XState < 4.0 **and** FP/Reactive ≥ 3.5 | Multi-domain fixes leakage. Investigate rank/scale for XState in v0.8. |
| 3.60 ≤ XState < 4.0 **and** (FP or Reactive) < 3.5 | Multi-domain insufficient at this scale. Reconsider recipe. |
| **XState < 3.60** (regression from v0.6) **and** FP/Reactive ≥ 3.5 | 32% XState share is too low; v0.8 with ≥40% XState. |
| **XState < 3.60** (regression from v0.6) **and** (FP or Reactive) < 3.5 | Recipe fundamentally wrong; pivot to RAG or rethink. |
| **Borderline**: any threshold crossed by ≤ 0.3 | Rerun the borderline arm under r=64 (if r=32) or wait for a second experimental anchor before committing. |

Tool calls must stay ≥ 2/3 in every branch (otherwise outcome (B) re-opens).

| Gate | XState | fp | rx |
|---|---|---|---|

| Idiom MUST-MATCH | `setup\(` | one of: `Effect\.gen`, `Effect\.succeed`, `Effect\.fail`, `Effect\.tap`, `Effect\.flatMap`, `Effect\.map`, `pipe\(`, `Layer\.`, `Schedule\.`, `Context\.Tag`, `makeContext`, `E\.chain`, `E\.right`, `E\.left`, `O\.chain`, `O\.some`, `O\.none`, `TE\.chain`, `TE\.right`, `TE\.left` | one of: `pipe\(`, `switchMap\(`, `combineLatest\(`, `mergeMap\(`, `forkJoin\(`, `BehaviorSubject`, `of\(`, `Subject`, `Observable<`, `ReplaySubject`, `retry\(`, `shareReplay`, `tap\(` |

## Risks / open questions for reviewer

These are the places I think I could be wrong:

1. **Domain proportion.** 47% XState / 29% FP / 21% RX might be wrong. An
   alternative view: since XState already works at v0.6 levels, reducing it
   to 30% and pushing FP/RX higher (~35% each) might fix the leakage faster
   without losing XState. I don't have strong evidence either way.

2. **Reusing v0.6 pairs.** If the fp/rx failure mode is "model has a strong
   XState prior," reusing v0.6's 440 XState-only pairs reinforces that prior.
   An alternative is to **re-synthesise XState pairs under the multi-domain
   prompt** so the synthesis signal is inherently domain-diverse from the
   start. Costs more subagent time.

3. **Verifier XState-leakage guard.** Rejecting fp/rx pairs that import from
   `xstate` is the right rejection at training time, but can we trust the
   verifier's gate set to catch all forms of XState-shaped code in an
   Effect-TS answer? A synthesis subagent could produce `setup`-shaped code
   in a file that happens to import from `effect` — the regex would miss it.
   Mitigation: the MUST-MATCH for fp includes Effect-specific tokens, so the
   synth would fail that gate instead. But it's not airtight.

4. **Single-arm grading bias.** The v0.6 grader was asked to "grade on
   quality alone, blinded to arm" but knew it was grading one arm. For v0.7,
   mixing v0.6 and v0.7 responses in a single blinded grading pass would
   give cleaner comparisons. Not implemented in the plan above — worth
   discussing.

5. **r=32 vs r=64.** Locked r=32 from v0.6 validation, but v0.7 pushes more
   aggressive syntactic shifts across three domains. r=64 may be worth a
   parallel arm if GPU time permits.

6. **Capability anchors count may need to scale.** 10% of 1400 = 140 anchor
   records, from 8 unique sources → 17 repetitions each. v0.6 had 6
   repetitions each. Unclear whether more repetition helps or overfits.

## Failure modes (per-codepath)

Each v0.7 codepath has one realistic production failure and coverage status:

- **FP/RX synthesis subagent iteration loop** (Phase C) — failure: subagent
  enters pathological verify-fix-verify loop and never produces a batch file.
  Coverage: subagent has explicit max-iteration clause; controller can re-dispatch.
  Silent-failure risk: low (reported in subagent status).
- **Per-domain tsc runtime** (Phase D) — failure: fp tsc finds Effect type
  mismatches across many probes, batch tsc call exceeds 10-min timeout.
  Coverage: v0.6 verifier already has `timeout=600` + `tsc:timeout` attribution.
  Silent-failure risk: low.
- **Rank A/B selection** (Phase E) — failure: r=64 crashes on the 5090 due
  to increased VRAM pressure (LoRA adapter weights scale with rank).
  Coverage: **MISSING** — plan doesn't pre-check VRAM headroom. Mitigation:
  run r=32 first; if r=64 OOMs, skip with a note in decision.md. Critical
  gap: **no**, acceptable because r=32 run still produces a shippable model.
- **Grader B isolation drift** (Phase G) — failure: grader B reads grader A's
  file despite the prompt forbidding it. Coverage: **MISSING** — no verify
  step that checks whether grader B's transcript mentioned grader A's file.
  Mitigation: include "did not read forbidden files" as a status in grader B's
  report. Critical gap: **no**, soft-fail at most.
- **Borderline band** (decision matrix) — failure: arms land at 3.7 on XState
  (within 0.3 of 4.0 threshold). Per post-review decision matrix, conclusion
  is "rerun the borderline arm under r=64 (if currently r=32)." Already
  covered by the A/B training plan.

**No critical silent-failure gaps.** One soft gap (grader isolation
verification) worth noting but not blocking.

## Worktree parallelization strategy

Sequential implementation — no worktree split recommended. Rationale: every
phase touches a shared artifact from the previous phase (seeds → verifier →
synth data → verified data → trained model → eval → grading → decision).
Parallel worktrees would only help the FP vs RX synthesis phases, and those
are already parallelized via concurrent subagents within one directory. Not
worth the worktree overhead.
