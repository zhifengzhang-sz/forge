## TL;DR

**Outcome: partial win.** v0.6 lifts qwen3:14b on the target domain from
**XState 1.60 → 3.60 (+2.00)** — a large absolute gain and the first time in
this project fine-tuning has produced a clear win over the base model on

- **Tool calls survive intact** (3/3 canary). Outcome (B) stays ruled out.
- **Non-target TypeScript domains regressed.** fp-02, fp-05, rx-02 produced

  mode the v0 arms did not exhibit.

## Why v0.6 worked where v0 didn't

v0 tested ~50 examples; the best arm cleared 1.80. v0.6 tested 440 verified
pairs — still below the published LoRA floor (~1k) — and cleared 3.60. That's
a 2× gain from roughly 9× more data, with:

- **Teacher quality**: every pair generated by Claude Opus 4.7 (5.00/5.00 on
  the target eval).
- **Compile verification**: every pair passes `tsc --strict` against `xstate@^5`
  before being admitted to the training set. v0's training pairs had no such
  gate and that's how v4 leakage crept in via hand-curation.
- **Higher LoRA capacity**: r=32 vs r=16, 3 epochs vs 2.

The ordering in the plan (teacher → verifier → rank/epochs) captured the real
levers. Loss curve was clean (1.33 → 0.27, no instability), grad norm stable
~0.3 throughout.

## Why fp/reactive regressed

The training set was 440 XState pairs + 49 capability anchors (8 unique,
~10% of training). Zero pairs targeting FP (fp-ts, Effect-TS) or RxJS. The
capability anchors include Python fib, bash one-liners, REST/GraphQL, refactor
planning, and tool-call exemplars — broad reasoning, but nothing TypeScript-
library-specific.

Result: when the model sees a TypeScript prompt that looks superficially like
"build something with an async/stream flavour", it reaches for what it was
just trained on — `setup({...}).createMachine({...})` — instead of the
library the prompt actually asked for.

Specific failures:
- `fp-02` (Effect-TS Layer): produced an XState machine with `fromPromise` actors.
- `fp-05` (Effect-TS Context.Tag / Layer.merge): produced `setup({...actors: { logger: fromCallback })`.
- `rx-02` (combineLatest): imported `{ combine, assign, fromPromise }` from xstate.

This is a training-distribution problem, not a fundamental limitation of
LoRA or the pipeline. The fix is to widen the training set so XState is no
longer the only TypeScript "shape" the model has learned to produce. Which is
exactly what v1 would do anyway.

## What v0.6 surfaced beyond the headline

1. **The ~1k published LoRA floor may be soft.** We got meaningful signal at
   440 verified pairs — below the floor cited in the v0.6 plan. This suggests
   that for a sharply-scoped syntactic correction (v4→v5 API shape), the floor
   may be closer to a few hundred than a few thousand. The curated / extracted
   v0 arms missed at 43/46 pairs; v0.6 hit at 440. The threshold for
   "enough" on this kind of task sits somewhere in between.

2. **XState-leakage is the shape of the forgetting curve.** v0 asked "does the
   model forget tool calls?" (it didn't). v0.6 shows the real forgetting mode
   is subtler: adjacent TypeScript-library tasks absorb XState priors. Capability
   anchors protect tool-call / python / prose but not in-language neighbours.

3. **Subagent-driven synthesis works.** The plan assumed direct Anthropic API
   calls; the user declined that path. Seven parallel Claude Code subagents
   produced 40 compile-verified pairs each in a single wave (~5–16 min wall
   time) with 100% survival. At ~40 pairs per dispatch, 1500 candidates would
   take ~8 waves or roughly 2 hours of subagent-bound wall time. This is a
   reusable capability for future iterations.

4. **Verifier pays its keep.** `v0.6/verify.py` batched tsc into one call per
   wave (~1s for 40 pairs). Per-candidate tsc would have cost ~5 s × 441 ≈
   37 min. The batched approach kept verification wall time under 3 s total
   across the whole run.

5. **Synthesis survival was ~100%, not the expected 54–72%.** Subagents
   iterated against the verifier before writing their batch files. The result
   was effectively pre-verified training data with no filter loss. This is
   better than the plan anticipated and changes the math for v0.7.

## What v0.6 was not

- **Not at floor.** 440 verified pairs is below the ~1k the plan called for.
  A v0.7 at 1500–2000 pairs would test whether XState continues climbing and
  whether broadening the training domain (adding fp/reactive pairs) fixes
  the regression.
- **Not blinded multi-grader.** Grading was done by one fresh subagent, not
  the multi-grader triangulation the plan methodology-debt section asked for.
  A second independent grader would strengthen the 3.60 number — realistic
  uncertainty band is maybe ±0.3.
- **Not rank-tuned.** We used r=32 across the board. r=64 was left on the
  table and remains a v0.7 lever if XState stalls out in the 3.5–4.0 band.
- **Not production-validated.** The decision.md numbers are eval-based. No
  real Claude-Code-in-the-loop validation on actual user tasks yet.

## Incident during v0.6

The machine lost power at **2026-04-18 23:22:54 EDT**, approximately 90 s
after the eval completed. Clean hardware-level cut — no kernel panic, no
thermal event, no OOM, no ACPI errors; last journal line is a routine
NetworkManager warning. Signature matches an RTX 5090 transient power spike
combined with CPU/disk activity during the Ollama model-load that followed
GGUF conversion. All outputs had been flushed to disk before the cut —
zero data loss.

Mitigation for future runs:
- `sudo nvidia-smi -pl 500` before heavy training to cap 5090 sustained draw.
- Serialize training → ollama model-load (don't overlap workload transitions).
- Verify PSU rating; 1200 W is the safe envelope for 5090 + high-core CPU.

## Recommended next steps

In priority order:

1. **Launch v0.7 with broader domain coverage.** Add ~500–800 fp-ts / Effect /
   RxJS pairs to the training set alongside another ~800 XState pairs. Goal:
   clear XState ≥ 4.0 without the fp/reactive regression. Same r=32, 3 epochs.
   Total pairs ~1300–1600 — at or slightly above the ~1k floor.

2. **Add a second independent grader** for v0.7. Re-grade v0.6 too, to firm
   up the 3.60 number and detect grader-specific bias.

3. **Stop treating capability anchors as a cure for forgetting.** 10% Python/
   bash/prose anchors protected tool calls and general reasoning but did not
   protect fp-ts / RxJS. The preservation signal has to come from the same
   domain-shape as what's at risk.

4. **Keep the verifier.** It held survival at 99.8% over 441 candidates,
   costs <3 s per full batch, and caught the one bad pair that would have
   contaminated training. This is the single highest-leverage piece of
   infrastructure the pipeline has.

5. **Ship v0.6 as a daily-driver for XState-only tasks.** It's a 3.60 on a
   narrow domain — not frontier, but strictly better than base qwen3:14b or
   qwen3-coder:30b on this specific use case. Worth using while v0.7 is
   being planned.

6. **Before v1**, validate that the teacher-verifier loop scales to multiple
   domains in one training run (not one domain at a time). v0.6 proved the
   loop works for XState; v0.7 will test whether it composes.

## What v0.6 was worth

Validates the pipeline: teacher-produced + compile-verified + r=32 LoRA on
qwen3:14b moves XState from 1.60 to 3.60. The premise of the project —
"a local 14B, properly fine-tuned, can close most of the gap to a frontier
model on a narrow domain" — is now partially supported by evidence, where
before v0.6 it was partially refuted.

Three findings are structural:
- The floor for LoRA-on-syntactic-correction is probably a few hundred
  verified pairs, not a few thousand.
- Forgetting in LoRA fine-tuning is domain-shaped, not capability-shaped.
  Capability anchors protect tool calls and general reasoning but not
  library-neighbour tasks.
- Subagent-driven synthesis with compile verification is a reusable pipeline
  that works without direct API spend.

The recommended immediate next move — v0.7 with multi-domain training —
is the same move v1 would have made anyway. v0.6 just earned the right to
take it on credit instead of speculation.
